# Theming — Dark Mode & Semantic Tokens (W13-A, 2026-05-15)

Phase Detector ships a 3-mode theme system (`system` / `light` / `dark`)
backed by a small set of semantic CSS custom properties. This doc explains
the philosophy, the migration story for legacy components, and how to debug
or extend the token set.

## Token philosophy

The token set is deliberately **semantic, not chromatic**. We do not expose
"purple-600" or "zinc-100" as tokens — those are *appearance*, not *meaning*.
Components ask for tokens like `--fg-primary` ("foreground for primary text"),
`--bg-elevated` ("a surface that floats above the page background"), or
`--accent` ("the call-to-action color"). The token-to-color mapping flips
inside `.dark { ... }`, so a component written against semantic tokens gets
correct dark-mode rendering for free.

This is the same pattern that Apple's HIG / Material You / shadcn-ui converge
on. It buys us three things:

1. **Components stay theme-agnostic.** No `if (isDark)` branching in JSX
   when a CSS variable can do the work.
2. **Future themes are cheap.** Adding a high-contrast accessibility theme
   is "add a new class to globals.css", not "fork every component".
3. **WCAG is verifiable per token.** The contrast ratio is a property of
   the token pair `(--fg-X, --bg-Y)`, not the component, so we audit once
   and trust everywhere.

Verified ratios (W13-A acceptance) on the **light** theme background `#FFFFFF`:

| pair                         | ratio   | WCAG       |
|------------------------------|---------|------------|
| `--fg-primary` on `--bg-base`  | 16.10:1 | AAA-normal |
| `--fg-secondary` on `--bg-base`|  7.95:1 | AAA-normal |
| `--fg-tertiary` on `--bg-base` |  4.83:1 | AA only    |
| `--accent` (#5B21B6) on white  |  8.85:1 | AAA-normal |

On the **dark** theme background `#0A0A0A`:

| pair                         | ratio   | WCAG       |
|------------------------------|---------|------------|
| `--fg-primary` (#FAFAFA)     | 18.71:1 | AAA-normal |
| `--fg-secondary` (#D4D4D8)   | 14.20:1 | AAA-normal |
| `--fg-tertiary` (#A1A1AA)    |  8.18:1 | AAA-normal |
| `--accent` (#A78BFA)         |  8.95:1 | AAA-normal |

`--fg-tertiary` is the one near-AA spot in light mode — reserved for
incidental chrome (timestamps, axis ticks). Body copy uses `--fg-secondary`
which clears AAA in both themes.

## Architecture

- `components/ThemeProvider.tsx` — React context. Reads `localStorage`
  key `phase_theme`, listens to `(prefers-color-scheme: dark)`, applies
  the `.dark` class on `<html>`. Provides `useTheme()` returning
  `{ theme, resolvedTheme, setTheme }`.
- `components/ThemeToggle.tsx` — segmented control (3 buttons: System /
  Light / Dark). Lives in TopNav next to LanguageSwitcher.
- `app/globals.css` — declares `:root` tokens + `.dark` overrides + a
  legacy compatibility layer that aliases the older `--text-primary` /
  `--bg-primary` names to the new tokens.
- `tailwind.config.ts` — exposes utilities `bg-base`, `text-fg-primary`,
  `border-token`, `accent`, etc. for new code. Legacy zinc utilities still
  work; we ship dark-mode overrides for the most common 8 (zinc-50/100/200,
  zinc-400/500/600/700/900, white) so unmigrated components flip
  automatically.

## Adding a new color token

1. Pick a **semantic name**. "What does this color *mean*?", not
   "what shade is it?". Example: `--surface-warning` (a warning-tinted
   surface), not `--yellow-50`.
2. Declare it in **both** `:root` and `.dark` in `globals.css`.
3. Run a contrast check against the token pair you intend to use it with
   (e.g. white text on warning surface). Target 7:1 for normal text,
   4.5:1 for large text / non-text UI.
4. (Optional) Expose as a Tailwind utility under
   `theme.extend.colors` in `tailwind.config.ts` — only if the token will
   be reused across many components. One-off colors can stay as inline
   `bg-[var(--surface-warning)]` arbitrary classes.
5. Document the token's intended use in this file.

## Per-component dark-mode debugging recipe

When a component looks broken in dark mode, walk the following decision tree:

1. **Is the broken element using a CSS variable?**
   Open DevTools → Computed → check the computed `color` / `background`.
   If it shows a literal hex (e.g. `#FFFFFF`), the component has a
   hardcoded color and needs to be migrated. Replace with `var(--bg-base)`
   or a Tailwind token utility (`bg-base`).
2. **Is the variable defined inside `.dark`?**
   If the variable is in `:root` but not in `.dark`, both themes resolve
   to the same value. Add the `.dark` override.
3. **Is the contrast ratio still ≥ 7:1?**
   Run the contrast snippet from `tests/e2e/test_dark_mode.py::
   test_wcag_aaa_body_contrast` against the broken element. If it drops
   below 7:1, brighten/darken the dark-mode token value.
4. **Is the component inside an SVG `<foreignObject>` / inline `style`?**
   Tailwind `dark:` variants don't reach inline `style={{}}`. Use the
   `useTheme()` hook and switch palettes in JS — see `SparkLine.tsx` /
   `PhaseTrajectoryChart.tsx` / `UniversalityAnalogueMap.tsx` for the
   pattern.
5. **Is it the legacy `--text-primary` / `--bg-primary` alias?**
   Those still work but resolve to the new tokens; the alias map lives at
   the bottom of `:root` in `globals.css`. If you change one alias,
   sweep all call sites.

## OG cards

Open-graph card images (`public/og/*.png`) are always **light-themed** —
they're previewed in social feeds where the surrounding context is set by
the consuming app (Twitter / Slack / iMessage), not our site. Generating
dark-themed OG cards would invert the legibility for users whose social
client is in light mode. If we ever ship dynamic OG image generation
(`app/opengraph-image.tsx`), it should force light tokens regardless of the
visitor's current theme.

## FOUC trade-off

We accept a ~30ms flash of unstyled content on first paint (the `.dark`
class is applied in a `useEffect`, not in a blocking `<script>` in `<head>`).
This avoids a CSP nonce / build-time script-injection complication. If the
flash becomes a real UX complaint, the standard fix is a tiny inline script:

```html
<script dangerouslySetInnerHTML={{ __html: `
  try { var t = localStorage.getItem('phase_theme');
        var d = t === 'dark' || (t === 'system' && matchMedia('(prefers-color-scheme: dark)').matches);
        if (d) document.documentElement.classList.add('dark');
  } catch(e) {} `}} />
```

Drop in `app/layout.tsx`'s `<head>` and the FOUC disappears at the cost of
a nonce/CSP review.
