# Structured logging + correlation IDs (W14-D, 2026-05-15)

This doc explains how the backend now emits logs, why we picked structlog,
how to trace a single request end-to-end, and where to look next.

## Why structlog (vs raw stdlib `logging`)

We already had a hand-rolled `JsonFormatter` in `services/observability.py`
that produced JSON-lines events. It worked, but every call site had to
remember the `extra={"fields": {...}}` boilerplate, fields silently
clobbered top-level keys, and there was no first-class contextvar binding.

structlog gives us four things stdlib alone doesn't:

1. **Free-form kwargs**: `log.info("ask.llm.start", model="x", kb_count=12)`
   becomes a JSON dict without ceremony. No `extra={"fields": {...}}`.
2. **Contextvars merge**: a `merge_contextvars` processor pulls anything
   bound via `structlog.contextvars.bind_contextvars(...)` onto every line
   in the current asyncio task — exactly what we need for correlation IDs.
3. **Unified pipeline for stdlib loggers**: `ProcessorFormatter` lets
   FastAPI / uvicorn / slowapi records share the same JSON shape as our
   own structlog calls. One stream, one schema.
4. **Idempotent configure**: re-calling `configure_logging()` doesn't
   stack handlers, which matters for our test fixtures.

## Correlation pattern

Every inbound HTTP request walks through `CorrelationIdMiddleware` first:

1. Read `X-Request-ID` header. If absent or malformed, generate a fresh
   UUID4 hex (32 chars, no dashes — easier to grep).
2. Bind the ID + path + method + tier onto module-level ContextVars in
   `logging_config.py`.
3. Run the request.
4. Echo the ID back as `X-Request-ID` on the response (so clients can
   quote it in support tickets / share-card screenshots).

Inside the request scope, **every** log line — whether from our code, from
uvicorn, from slowapi's 429 path, or from a third-party library — gets
`request_id`, `path`, `method`, and (when known) `tier` injected by the
`_inject_correlation_ctx` processor.

Critical endpoints emit explicit lifecycle events so a single grep on the
ID reconstructs the entire pipeline:

| Endpoint | Events |
|---|---|
| `/api/ask/stream` | `ask.request` → `http.request` → `ask.llm.start` → `ask.llm.complete` → `ask.response` |
| `/phase/api/companies` | `phases.fetch` (with `ticker_count`) |
| `/api/checkout/mock` | `checkout.success` / `checkout.declined` (with tier + amount) |

## How to query

The rotating log file lives at `web/backend/logs/server.jsonl` (rotates
at 100 MB, 7 backups). To trace one request:

```bash
# All events for request abc12345
jq 'select(.request_id == "abc12345")' web/backend/logs/server.jsonl

# All slow LLM calls (> 5s)
jq 'select(.event == "ask.llm.complete" and .latency_ms > 5000)' \
  web/backend/logs/server.jsonl

# All declined checkouts in the last 100 lines (no jq stream support)
tail -100 web/backend/logs/server.jsonl | jq 'select(.event == "checkout.declined")'

# Group by tier
jq -r '.tier // "anon"' web/backend/logs/server.jsonl | sort | uniq -c
```

## Admin tail endpoint

`GET /api/admin/logs/tail?n=200&filter=ask` returns the last N JSON lines
from the rotating file, with optional substring (`filter`) and level
filters. Admin tier (`X-API-Key`) required. The implementation tails by
*line* (not bytes) so we never straddle a rotation boundary.

Useful when SSHing into the box is overkill — e.g. quickly checking how a
single user's recent ask requests went:

```bash
curl -H "X-API-Key: $ADMIN_KEY" \
  "https://structural.bytedance.city/api/admin/logs/tail?n=50&filter=ask.llm"
```

## What's deliberately out of scope

- **Shipping to Loki / OpenSearch**: stdout + rotating file covers the
  current ops surface (single VPS + systemd journal). When we move
  multi-node, swap the file handler for a Loki HTTP push or a Promtail
  sidecar — the structlog pipeline doesn't need to change.
- **Sentry / error grouping**: the existing `services/observability.py`
  Sentry hook stays in place; structlog logs are not auto-forwarded to
  Sentry (would burn quota on info-level events). Errors raised inside
  request handlers still hit Sentry via FastAPI's exception path.
- **PII scrubbing**: we log query *length* not the query itself for
  `/api/ask` so user prompts don't end up in logs. If/when we add full
  query logging for debug, gate behind `STRUCTURAL_LOG_QUERIES=1` env
  and ensure logs are encrypted at rest.

## Operational knobs

| Env var | Purpose | Default |
|---|---|---|
| `STRUCTURAL_LOG_LEVEL` | Root log level | `INFO` |
| `STRUCTURAL_LOG_FILE` | Override rotating file path | `web/backend/logs/server.jsonl` |
| `STRUCTURAL_ENV` | Stamped as `env` field on every line | unset |
